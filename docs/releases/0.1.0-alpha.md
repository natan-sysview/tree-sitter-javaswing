# 0.1.0-alpha.1 Release Notes

`tree-sitter-javaswing` is a controlled fork of `tree-sitter-java` that keeps Java parsing intact while adding named nodes for official Swing and AWT frontend structure.

This alpha is intended for grammar review, downstream parser-lab validation, and early extraction experiments. It should be published as a GitHub pre-release until downstream users have validated the tree shape.

## Highlights

- Adds official Java Swing and AWT frontend nodes without relying on client naming conventions.
- Covers common UI construction patterns such as screens, components, fields, `initComponents`, listeners, layout wiring, dialogs, file choosers, table/list/combo models, root panes, tabbed panes, split panes, scroll panes, actions, timers, workers, focus, transfer handlers, and UI-thread invocation.
- Ships synthetic public corpus tests instead of private source-derived examples.
- Includes generated public node documentation in `docs/nodes.md`.
- Includes npm packaging and CI checks for generation, corpus tests, queries, public example parsing, and dry-run package contents.
- Includes a public synthetic showcase under `examples/`.
- Runs public CI across Linux, macOS, and Windows.

## Known Limits

- The grammar does not model client-specific subclasses or proprietary GUI helper frameworks.
- Third-party layout DSLs such as JGoodies `CellConstraints` and TableLayout profiles are intentionally excluded from the official grammar.
- Broad calls such as `add(...)`, `setModel(...)`, `getModel()`, `setSelectedIndex(...)`, and `setResizable(...)` remain generic when the receiver type cannot be proven from local syntax.
- The grammar does not perform semantic inheritance, import-aware type resolution, or symbol-table analysis.

## Release Checklist

Before tagging this release publicly:

1. Set the final GitHub repository URL in `tree-sitter.json`.
2. Add the same repository URL to `package.json`.
3. Update `docs/parser-list.md` with the public repository details.
4. Copy only the public payload into a clean standalone repository.
5. Run `npm ci`.
6. Run `npm run ci:check`.
7. Inspect the package dry-run output for private files or paths.
8. Confirm the release workflow created the GitHub pre-release.
9. Publish npm with the manual `Publish npm` workflow or with local `npm publish --tag alpha` only after credentials and CI are ready.
